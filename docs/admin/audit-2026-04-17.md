## 0. TL;DR — why operator says "still not showing well"

Three concrete bugs, plus a category of zero-state ugliness:

1. **Top spenders table renders rows with $0 revenue + 0 tokens** because the query has no activity threshold. Looks like the table is broken when really it's just listing inactive users. **High** severity — most jarring single defect.
2. **Revenue card shows `$0` instead of "Stripe not wired"** when Stripe isn't configured but the (existing, empty) `stripe_invoices` table returned 0 cents. The empty-state branch only fires on `null`, not on `configured=false`. **High** severity.
3. **Sparkline renders a near-flat line with two 1-event spikes** at 30-day scale. Technically correct but visually weak; reads as broken because there's so much white space. **Medium** severity.
4. **Tables on `/admin/requests` and `/admin/api-keys` are entirely empty.** The empty-state copy exists ("No requests in the last period.") but it's a single sentence — directive said it should be intentional ("No requests yet — your first API call will appear here"). **Medium** severity.
5. **Skeleton-then-empty-state layout shift.** When real data is empty, the skeleton rows render briefly then collapse into a one-line message — gives the page a "twitchy" feel. **Low** severity.

The rest of this doc is the systematic audit.

## 2. Issue list, prioritized

### S1 — Severity HIGH (blocks "shows well")

#### S1.1 — Top spenders shows $0 / 0-token rows

**Where:** `pages/admin/index.jsx → TopSpendersCard`, fed by `pages/api/admin/overview.js → topSpenders` query (`LIMIT 5` ordered by revenue then tokens).

**Live behavior:** With 0 rows of `stripe_invoices` and 0 rows of `usage_logs`, the query returns up to 5 users sorted by `(0, 0)`, so it picks the first 5 user records. Renders as a table where every row reads `email | $0 | 0`. Looks broken.

**Fix:** Either (a) add `WHERE` clause to filter users with revenue or tokens > 0, OR (b) JS-side filter. (b) keeps the API stable. Then if the filtered list is empty, show the existing 'No spender activity in the last 30 days' empty state.

**Effort:** XS. 5–10 lines.

---

#### S1.2 — Revenue card shows `$0` instead of "Stripe not wired"

**Where:** `pages/admin/index.jsx → RevenueKpi`. Empty-branch:

```js
{cents == null ? '—' : formatCurrency(cents, currency)}
```

**Live behavior:** `revenue.todayCents = 0` (number, not null) so the empty branch never fires. Card shows `$0` + "MTD $0" sub. Misleading — implies a real customer paid $0 today.

**Fix:** Promote the `stripe.configured` flag through to RevenueKpi. When not configured, render as `'—'` + "Stripe not wired" sub regardless of cents. When configured but cents=0, render as `$0` + "no revenue today" or similar.

**Effort:** XS. 5–10 lines.

---

#### S1.3 — Empty-states across `/admin/requests` and `/admin/api-keys` are too sparse

**Where:** `pages/admin/requests.jsx` empty-branch is `<p>No requests recorded yet.</p>`. `pages/admin/api-keys.jsx` is `<p>No API keys have been created yet.</p>`. Both are single sentences with no illustration or CTA hint.

**Fix:** Replace with structured empty-state — title + body sentence + (optional) CTA link to docs. Per directive's example: "No requests yet — your first API call will appear here" with a link to `/docs/quickstart`. Reuse the `EmptyState` pattern from `design/DESIGN_SYSTEM.md §7.10` (component doesn't exist yet — see S2.5).

**Effort:** S. ~20 lines per page.

### S2 — Severity MEDIUM (visible polish)

#### S2.1 — Sparkline reads weak with mostly-zero data

**Where:** `pages/admin/index.jsx → Sparkline` component.

**Live behavior:** With `signupsByDay` being 28 zeros + 2 ones over 30 days and `max = Math.max(1, ...counts)` = 1, the SVG path is a near-flat line at the bottom of the 28px box with two tiny upward spikes. Visually sparse.

**Fix options:**
- (a) Bake a baseline gradient under the line so the area visualizes the "no activity" period
- (b) When `max ≤ 2 OR points.length - nonZero >= 25`, render a "Limited activity in last 30d" muted text instead of the sparkline
- (c) Auto-scale Y so even a single 1-event day stretches to half-height (already does this — max=1 fills full height)

I'd ship (b) plus (a) — the muted text is the honest state when activity is below the visual threshold; the gradient helps when there's actual signal.

**Effort:** S. ~30 lines.

#### S2.2 — `/admin/users` "Customers" view: per-user 30d revenue column shows `—` for everyone

**Live behavior:** All users show `—` (correct, since no Stripe data). But the entire column is dashes, taking horizontal space without value. Not strictly broken — just weak when nothing is paying.

**Fix:** Either (a) hide the 30d revenue column entirely when `summary.stripeColumnsAvailable === false` OR all users have `stripe30dCents === 0`. Same for MRR column. OR (b) keep columns but visually de-emphasize when uniformly empty.

I'd ship (a) — drop both Stripe-derived columns when summary indicates Stripe is unwired. The Customers table reads cleaner.

#### S2.3 — Skeleton → empty-state layout shift

**Where:** Multiple cards. Skeleton renders for ~150ms during fetch, then collapses to a one-line empty message — visible "twitch."

**Fix:** Two options:
- (a) Pin a min-height on each card body so the empty message takes the same vertical space as the skeleton would
- (b) Show the empty-state immediately when the API returns `[]` (no skeleton flash) — but this requires the loader to know whether the eventual result will be empty, which it can't

(a) is the easy + correct fix. Apply to `RecentRequestsSkeleton`, `TopSpendersCard`, `StripeStatusCard`, `AwsStatusCard`.

**Effort:** XS. ~10 lines of CSS.

#### S2.4 — Date formatting inconsistent across cards

**Where:** Various.

**Live behavior:**
- `db-page-subtitle` shows "Updated HH:MM" (only time of day, no date) — fine for a same-session view, weird if cached.
- TopSpenders table doesn't show dates.
- Recent requests table shows `new Date(r.createdAt).toLocaleTimeString()` — local time of day, no date.
- Customers view: `lastRequestAt` shown as `new Date(...).toLocaleDateString()` — date, no time.

Per directive: "Date formatting consistent (relative time for recent, absolute for old)."

**Fix:** Centralize in a `formatTimeSmart(iso)` helper:
- < 60s ago → "just now"
- < 1h → "Nm ago"
- < 24h → "Hh ago"
- same calendar day → "HH:MM"
- < 7d → "EEE HH:MM"
- otherwise → "MMM d, yyyy"

Apply to `db-page-subtitle`, recent-requests table, customers `lastRequestAt`, system-status webhook timestamps.

**Effort:** S. ~30 lines.

## 5. Design recommendation for in-app key entry (Step 3)

Operator asked for a decision and a comment.

**Recommendation: Option B — Form posts to backend → Vercel REST API sets env var → triggers redeploy.**

**Why B over A:**
1. **Less code change.** Every Stripe / AWS consumer (`lib/stripe.js`, `pages/api/webhooks/stripe.js`, `lib/aws.js`, `services/api-gateway/src/lib/inference/AwsInferenceBackend.js`) reads `process.env.X` today. Option A requires a runtime DB lookup with a cache, plus a refactor of every consumer. Option B is purely additive — the consumers don't change at all.
2. **Vercel handles encryption + access logging.** Their env-var store is encrypted at rest, and changes are auditable in the Vercel dashboard. Reusing it avoids re-implementing key management with master-key rotation.
3. **Aligns with current operator workflow.** Operator already sets env vars manually in Vercel Settings → Environment Variables. Option B just adds a UI shortcut for that exact action.
4. **Test connection works either way** — for B, the form input is sent to the backend, which calls `Stripe.customers.list({limit:1})` or `STSClient.getCallerIdentity()` with the *unsaved* key. Returns success / error; only on success does the form let the user click "Save & redeploy".

**Tradeoffs accepted:**
- Each save triggers a redeploy (~30-90s deploy). Acceptable; integration changes are infrequent.
- `VERCEL_TOKEN` lives in Vercel envs, accessible to admin endpoint code. Add a server-only env var (`CLOUDACH_INTEGRATIONS_TOKEN`) gating the integrations endpoints; only this token can call the Vercel API. Or simpler: rely on `requireAdmin` middleware (already gates by DB role).

**What this commits the next pass to:**
- New page `pages/admin/integrations.jsx` with two cards: Stripe, AWS.
- New API `pages/api/admin/integrations/[provider].js` — POST { keyName, value } updates the Vercel project env var via REST.
- New API `pages/api/admin/integrations/[provider]/test.js` — POST { keyName, value } runs a probe (Stripe customers.list / AWS STS getCallerIdentity) and returns `{ ok: bool, message: string }`.
- Audit log entry per save in `audit_logs` table (already exists per migration).
- Triggers a fresh Vercel deploy via `POST /v13/deployments` after each save.
- No new DB migration needed. (Option A would have required `infra/db/013_system_settings.sql` + master encryption key.)

I'll comment on the issue with the decision and proceed when this audit doc merges.
